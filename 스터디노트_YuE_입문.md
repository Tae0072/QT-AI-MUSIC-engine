# 스터디 노트 — YuE 작업에서 배우는 개념 (입문자용)

작성일: 2026-06-07
이번 작업에서 나온 개념들을 입문자 눈높이로 정리했습니다. "왜 그런지"를 이해하는 데 초점을 맞췄어요.

---

## 1. 가상환경(venv)과 "베이스 파이썬"
- **venv**: 프로젝트 전용 파이썬 패키지 공간. 프로젝트마다 버전 충돌 없이 라이브러리를 따로 관리.
- 중요한 점: venv 안의 `python.exe` 는 **완전한 파이썬이 아니라**, 설치돼 있는 "베이스 파이썬"(여기선 3.10.11)을 **빌려 쓰는 얇은 연결**입니다.
- 그래서 **베이스 파이썬을 지우면 venv가 통째로 안 돕니다.** (이번에 앱이 안 뜬 원인 ①)
- 교훈: 특정 버전(3.10)에 의존하는 프로젝트는 그 버전을 지우지 말 것.

## 2. 절대경로 vs 상대경로 (드라이브 문자 함정)
- `E:\ai엔진\...` 처럼 **드라이브 문자를 코드에 박아두면**, 폴더를 옮기거나 드라이브 문자가 바뀌면 깨집니다. (원인 ②)
- 해결: **스크립트 자기 위치 기준**으로 경로를 계산(`__file__` 활용). 어디로 옮겨도 안 깨짐.
- 교훈: 경로는 가능하면 "기준점 + 상대경로"로.

## 3. 실행 파일들: VBS / PowerShell / BAT
- **.bat**: 옛날 명령 프롬프트 스크립트. 간단한 일에.
- **.vbs**: 창(콘솔) 없이 다른 걸 조용히 실행할 때 유용. (그래서 런처로 사용)
- **.ps1 (PowerShell)**: 강력. 창 만들기(WinForms), 프로세스 제어, 네트워크 등 가능. (로딩바를 여기서 만듦)
- 콘솔 창 숨김 실행: `powershell -WindowStyle Hidden -File ...` + VBS의 `sh.Run cmd, 0, False`.

## 4. 글자 인코딩 (한글 깨짐의 원인)
- 컴퓨터는 글자를 숫자로 저장하는데, 그 규칙이 **인코딩**입니다(UTF-8, UTF-16, 등).
- **BOM**: 파일 맨 앞의 작은 표식. "이 파일은 UTF-8/UTF-16이야"라고 알려줌.
- 이번 교훈: PowerShell(.ps1)은 한글 있으면 **UTF-8 BOM**, WSH(.vbs)는 **UTF-16LE BOM**으로 저장해야 한글이 안 깨짐. 안 그러면 `□□□` 같은 깨짐 발생.

## 5. 포트 · localhost · 서버/클라이언트 · gradio
- **서버**: 백그라운드에서 도는 프로그램(여기선 python gradio). **클라이언트**: 그걸 쓰는 쪽(브라우저).
- **포트(7862)**: 한 컴퓨터 안에서 프로그램을 구분하는 "방 번호". `http://localhost:7862` = 내 컴퓨터의 7862번 방.
- **gradio**: 파이썬으로 웹 UI를 쉽게 만들어 주는 도구. AI 모델을 브라우저에서 쓰게 해줌.
- "포트가 LISTEN 중" = 서버가 준비돼 접속을 받는 상태. 우리 로딩바는 이걸 확인해 "준비됨"을 판단.

## 6. 브라우저 "앱 모드" + watchdog (창 닫힘 감지)
- 일반 브라우저 **탭**은 여러 개가 한 프로세스라, 특정 탭이 닫혔는지 외부에서 알기 어려움.
- **앱 모드**(`chrome --app=URL --user-data-dir=전용폴더`): 탭/주소창 없는 **독립 창 = 독립 프로세스**로 띄움.
- **watchdog**: 그 프로세스가 살아있는지 지켜보다, 닫히면 서버를 종료. 그래서 "앱 창 닫으면 서버도 종료"가 가능.
- **프로세스 트리 종료**: `taskkill /PID x /T /F` 의 `/T` 는 자식 프로세스까지 함께 종료. (gradio가 자식 python을 띄우므로 필요)

## 7. GPU / VRAM / 양자화 — "왜 느린가"의 핵심
- **CPU vs GPU**: AI 모델 연산은 GPU가 훨씬 빠름. GPU가 놀고 CPU로 돌면 매우 느림.
- **VRAM**: GPU 전용 메모리(여기 16GB). 모델이 VRAM에 다 들어가야 GPU가 빠르게 돔.
- **모델 크기**: YuE stage1 은 **7B(70억 파라미터)**. 일반 정밀도(bf16)면 약 14GB → 16GB에 빠듯해서 다 못 올림.
- **양자화(quantization)**: 모델을 더 작은 숫자형(int8)으로 줄여 VRAM에 맞춤. 단, 이 환경의 int8 연산은 **느림(~8 token/s)** → 그래서 GPU를 써도 안 빨라짐.
- **mmgp / profile**: 이 프로그램은 메모리 절약용 라이브러리(mmgp)와 profile(1~5)로 모델 배치를 조절. 16GB에선 어떤 profile이든 빠르게 만들기 어려움.
- 결론: **16GB VRAM + 7B 모델 = 느림이 거의 정해져 있음.** 빠르게 하려면 더 작은 모델이나 VRAM 큰 GPU.

## 8. YuE 노래 생성의 2단계 구조 (★ "브금만이 왜 안 빨라지나")
1. **stage1 (7B, 가장 느림)**: 가사·장르를 받아 **보컬 트랙과 악기 트랙을 한 번에 같이** 생성. ← 시간의 대부분.
2. **stage2 (1B)**: 각 트랙을 다듬음(정제).
3. **보코더**: 저음질을 고음질로 업샘플.
4. **합성/후처리**: 트랙을 섞고 최종 파일 생성.
- **핵심:** stage1이 보컬·악기를 **함께** 만들기 때문에, "브금만"을 원해도 stage1 시간은 똑같습니다. 우리가 한 "브금만" 최적화는 뒤쪽 작은 단계(보컬 stage2·보코더)만 건너뛰는 것이라, **전체 시간은 크게 줄지 않습니다.** (보컬을 결과에서 빼는 효과는 확실히 있음)

## 9. winget (Windows 패키지 관리자)
- 명령 한 줄로 공식 소프트웨어를 설치하는 OS 기본 도구. (Python 3.10.11 재설치에 사용)
- 예: `winget install Python.Python.3.10 --override "/quiet TargetDir=..."` 처럼 설치 옵션도 지정 가능.

## 10. 디버깅에서 배운 점
- "프로세스 조회(CIM CommandLine)"는 가끔 **거짓 0(없음)** 을 반환 → 포트/HTTP 같은 **다른 신호로 교차 확인**이 안전.
- 서버가 **로그 없이 조용히 사라지면** 대개 외부에서 강제 종료(`taskkill /F`)된 것 — 우리 경우 watchdog가 앱 창 닫힘에 반응한 것.
- 메모리 부족 신호: `WinError 1455 (페이징 파일이 너무 작음)` → 여러 모델을 동시에 RAM에 올릴 때 발생.

### 한 줄 요약
앱이 안 뜬 건 **사라진 Python 3.10 + 잘못된 드라이브 경로** 때문이었고(둘 다 해결), 느린 건 **16GB GPU에 7B 모델이 빠듯해 느린 int8로 도는 구조적 한계** 때문입니다. "브금만"은 보컬은 빠졌지만(해결), 가장 느린 stage1이 보컬·악기를 함께 만들어서 시간은 크게 줄지 않습니다.
