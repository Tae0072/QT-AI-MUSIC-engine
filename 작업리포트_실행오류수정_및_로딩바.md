## 1. 한 줄 요약

"앱이 안 뜨던" 원인 **2가지**를 고치고, 실행할 때 화면 상단에 **로딩바**가 뜨도록 만들었습니다. 또한 **앱 창을 닫으면 백그라운드 서버도 자동으로 종료**되게 했습니다. 이제 `YuE_실행.vbs`를 더블클릭하면 → 상단 진행바가 표시되고 → 서버 준비가 끝나면 → **전용 앱 창**이 열리고 → 그 창을 닫으면 서버가 함께 꺼집니다.

## 2. 무엇이 문제였나 (원인 2가지)

### 원인 ① 파이썬(Python 3.10)이 사라짐
- `venv310` 가상환경은 **Python 3.10.11** 이라는 본체를 빌려 씁니다.
- 그 본체가 있던 폴더(`...\AppData\Local\Programs\Python\Python310`)가 **통째로 비어 있었습니다.** (PC에 3.13, 3.14만 남아 있었음)
- 그래서 `venv310\Scripts\python.exe` 가 **아예 실행되지 않았고**, VBS가 서버를 "숨김(백그라운드)"으로 띄우다 보니 에러 화면도 안 보이고 그냥 안 떴습니다.
- 다행히 venv 안의 큰 패키지(torch 4.4GB 등)는 멀쩡해서, **본체 3.10만 되살리면** 바로 동작하는 상태였습니다.

### 원인 ② 모델 경로가 잘못된 드라이브(E:)를 가리킴
- 프로젝트는 `D:\ai엔진\YuEGP` 에 있는데, 코드는 모델을 **`E:\ai엔진\...`** 에서 찾고 있었습니다.
- 실제 모델은 `D:` 에 정상적으로 있었는데도, 잘못된 E: 경로 때문에 서버가 시작 직후 멈췄습니다.
  ```
  OSError: Incorrect path_or_model_id: 'E:/ai엔진/YuEGP/models/YuE-s1-7B-anneal-jp-kr-cot'
  ```

## 3. 어떻게 고쳤나

### 고침 ① Python 3.10.11 재설치
- Windows 기본 패키지 관리자(winget)로 **공식 python.org 3.10.11** 을 원래 경로에 그대로 재설치했습니다.
- PATH·파일연결·py런처는 건드리지 않는 **최소 설치**라, 기존 3.13/3.14 환경에 영향이 없습니다.
- 결과: `venv310\Scripts\python.exe --version` → `Python 3.10.11` 정상 동작 확인.

### 고침 ② 모델 경로를 "자동 계산" 방식으로 변경
- 파일: `inference/gradio_server.py` (백업: 같은 폴더 `gradio_server.py.bak6_...`)
- 드라이브 문자(D:/E:)가 또 바뀌어도 깨지지 않도록, **스크립트 위치를 기준으로** 모델 폴더를 찾게 바꿨습니다.

  변경 전:
  ```python
  args.stage1_model="E:/ai엔진/YuEGP/models/YuE-s1-7B-anneal-jp-kr-cot"
  args.stage2_model="E:/ai엔진/YuEGP/models/YuE-s2-1B-general"
  ```
  변경 후:
  ```python
  _PROJECT_DIR = os.path.dirname(os.path.dirname(os.path.abspath(__file__)))  # YuEGP 폴더
  _MODELS_DIR  = os.path.join(_PROJECT_DIR, "models")
  ...
  args.stage1_model = os.path.join(_MODELS_DIR, "YuE-s1-7B-anneal-jp-kr-cot")
  args.stage2_model = os.path.join(_MODELS_DIR, "YuE-s2-1B-general")
  ```

## 4. 로딩바 (새로 추가)

### 동작
1. `YuE_실행.vbs` 더블클릭 (콘솔 창 안 뜸)
2. 화면 **상단 중앙**에 진행바 표시: "♪ AI 노래 생성기를 준비하고 있어요 / 모델을 불러오는 중입니다 (최대 1~2분)" + 경과 시간
3. 서버 준비가 끝나면 → 진행바 100% → **전용 앱 창**(Chrome 앱 모드, 탭·주소창 없는 깔끔한 창)으로 앱이 열림 → 진행바 자동 닫힘
4. **그 앱 창을 닫으면 백그라운드 서버(python)도 함께 자동 종료** (창 닫힘 약 2초 내 종료 확인)
5. 이미 서버가 떠 있으면 → 진행바 없이 앱 창만 바로 열림 (중복 실행 방지)
6. 준비 도중 진행바의 **✕** 를 누르면 시작을 취소하고 서버도 종료
7. 시간이 오래 걸려도 끝까지 기다림(중간에 "조금 더 걸려요" 안내)

### 종료 방식이 동작하는 원리 (참고)
- 일반 브라우저 "탭"으로 열면 그 탭만 닫혔는지 알 수 없어 서버 종료를 감지할 수 없습니다.
- 그래서 앱을 **전용 프로필의 Chrome 앱 창**으로 띄웁니다. 이 창은 독립된 프로세스라, 창이 닫히면 스크립트가 그걸 감지해(watchdog) 7862 포트의 서버를 종료합니다.
- 별도 종료 도구(`YuE_종료.bat`)도 그대로 사용할 수 있습니다.

### 관련 파일
| 파일 | 역할 |
|------|------|
| `YuE_실행.vbs` | 진입점. 로딩바 스크립트를 콘솔 없이 실행 (원본은 `YuE_실행.vbs.bak_원본_...` 으로 백업) |
| `YuE_로딩바.ps1` | 서버 시작 + 상단 진행바 표시 + 준비되면 브라우저 열기 |
| `YuE_종료.bat` | (기존) 서버 종료 |

## 7. 추가 작업: "브금만 만들기"에 보컬이 섞이던 문제 (2026-06-07)

### 확인된 문제
- "브금만 만들기"를 켜도 최종 곡에 **보컬이 섞여** 있었음. 출력 합성 코드가 항상 `악기 + 보컬`로 합쳤음(최근 수정에서 빠진 회귀).
- 게다가 보컬 트랙을 똑같이 다 생성·후처리해서 **시간도 줄지 않았음.**

### 수정 내용 (`inference/gradio_server.py`, 백업: `gradio_server.py.bak7_brgm_...`)
1. 브금만 모드면 stage2·보코더에서 **보컬 트랙(vtrack) 제외** → 보컬 후처리 시간 절약 + 보컬 없음
2. 16kHz·44kHz 합성 모두 **악기 트랙만** 출력하도록 분기 복원
3. stage2 트랙 라벨을 파일명(`_itrack`) 기준으로 정정

### 검증
- `py_compile` 통과, 새 코드로 서버 정상 부팅
- 실제 브금만 생성 시 stage2가 **악기 1트랙만 처리(진행표시 0/1)** 되는 것 확인 → 보컬 제외 동작 OK
- (최종 렌더 파일은 아래 메모리 문제로 생성 도중 중단되어 끝까지 확인하지 못함)

### 별도로 발견한 속도·메모리 문제
- 기존 `--profile 3` 은 모델을 **시스템 RAM에 올려 GPU로 스트리밍**함 → GPU를 거의 안 쓰고(사용률 ~0%) 느림. RAM도 많이 써서 다른 앱(게임 등)과 겹치면 `WinError 1455(페이징 파일 부족)` 로 서버가 죽을 수 있음.
- GPU는 RTX 4070 Ti SUPER 16GB, CUDA 정상인데 거의 놀고 있었음.

## 8. GPU 속도 최적화 — 시도했으나 되돌림 (2026-06-07)

### 시도
- profile 3 에 VRAM 예산(budget=10000)을 주어 양자화 모델을 VRAM에 상주시켜 봄.
- 실측: 생성 중 GPU 95~97% 사용, VRAM ~9.6GB, RAM 17GB 확보. 짧은 곡(1구간)은 완주.

### 되돌린 이유
- **빨라지지 않았다:** 16GB VRAM이라 7B를 bf16로 못 올리고 **int8 양자화가 강제**되는데, 이 int8 연산이 이 환경에서 느림(약 8 token/s). GPU를 쓰든 안 쓰든 이 병목이 동일.
- **긴 곡에서 메모리 위험:** VRAM 상주는 긴 곡(여러 구간)에서 VRAM 초과(OOM) 위험을 더함.
- 그래서 **원래 설정(profile 3 기본)으로 복원**해 안정성을 우선함. (현재 코드엔 budget 변경 없음)

### 속도에 대한 솔직한 결론 (중요)
- **"브금만"이 안 빨라지는 진짜 이유:** YuE는 stage1(7B, 가장 느린 단계)에서 **보컬과 악기를 한 덩어리로 함께 생성**한다. 즉 보컬을 원하든 안 원하든 stage1 시간은 동일하다. "브금만"은 그 뒤의 작은 단계(보컬 트랙 stage2·보코더)만 건너뛰므로 **전체 시간은 크게 줄지 않는다.**
- **하드웨어 한계:** 16GB VRAM + 7B 모델 → 느린 int8 강제. 60초/3구간 곡은 토큰이 매우 많아 수십 분이 걸린다.
- **설정으로는 큰 속도 개선이 불가.** 근본 해결은 ① 더 짧은 곡, ② 더 작은 모델, ③ VRAM 24GB+ GPU.

### 지금 당장 빠르게 쓰는 법
- **길이(초)를 줄이세요.** 시간은 길이·구간 수에 거의 비례합니다. 예: 60초(3구간) → 30초(약 1구간)면 대략 1/3 시간.
- 무거운 앱(게임 등)은 닫고 생성하면 더 안정적입니다.
- 브금만 모드 자체는 정상 동작합니다(보컬 없음 + 보컬 후처리 시간 절약).
