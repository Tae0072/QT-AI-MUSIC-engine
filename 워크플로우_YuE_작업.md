## 작업 환경
- 대상 프로그램: `D:\ai엔진\YuEGP` (YuE 기반 AI 노래 생성기, gradio 웹앱, 포트 7862)
- 실행 구조: `YuE_실행.vbs` → `YuE_로딩바.ps1` → `venv310` 파이썬으로 `inference/gradio_server.py` 구동
- 하드웨어: RTX 4070 Ti SUPER(16GB), RAM 31GB

## 1단계 — "앱이 안 뜸" 진단
- **무엇을:** 실행해도 브라우저에 앱이 안 뜨는 원인 추적.
- **어떻게:** 7862 포트/프로세스 점검 → 서버를 창 보이게 직접 실행해 에러 캡처.
- **발견 1:** `venv310` 의 베이스 **Python 3.10.11 이 PC에서 사라짐** → venv 파이썬이 실행조차 안 됨.
- **발견 2(이후):** 모델 경로가 `E:\` 로 하드코딩(실제는 `D:`).
- **확인:** `pyvenv.cfg` 의 home 경로가 빈 폴더를 가리킴, 포트 LISTEN 안 됨.

## 2단계 — Python 3.10.11 복구
- **어떻게:** Windows 패키지 관리자 `winget` 으로 **공식 python.org 3.10.11** 을 원래 경로에 최소 설치(PATH·파일연결 안 건드림).
- **확인:** `venv310\Scripts\python.exe --version` → `Python 3.10.11`, torch/gradio import 정상.

## 3단계 — 모델 경로 수정
- **무엇을:** `gradio_server.py` 의 `E:/...` 하드코딩 2곳을 **스크립트 위치 기준 자동 계산**으로 변경.
- **왜:** 드라이브 문자(D:/E:)가 바뀌어도 안 깨지게.
- **확인:** 서버 부팅 → 모델 로딩 완료 → 7862 **HTTP 200**, 브라우저에 앱 표시. (백업: `gradio_server.py.bak6_*`)

## 4단계 — 상단 로딩바 제작
- **무엇을:** 실행 시 화면 상단 중앙에 진행바 표시(서버 준비~브라우저 열림까지).
- **어떻게:** `YuE_로딩바.ps1`(PowerShell + WinForms) 신규 작성, `YuE_실행.vbs` 는 이 스크립트를 콘솔 없이 실행하는 얇은 런처로 교체.
- **주의점:** 인코딩(PS1=UTF-8 BOM, VBS=UTF-16LE BOM)으로 한글 깨짐 방지. Linux측에서 원자적으로 작성해 에디터 충돌(파일 잘림) 회피.
- **확인:** 화면 캡처로 상단 바 표시 확인, 준비되면 자동으로 브라우저 열림.

## 5단계 — "앱 창 닫으면 서버도 종료"
- **무엇을:** 앱 창을 닫으면 백그라운드 서버(python)도 같이 종료.
- **왜 단순 탭으로는 안 되나:** 일반 브라우저 탭은 "닫혔는지" 알 수 없음.
- **어떻게:** 앱을 **Chrome 앱 모드 + 전용 프로필**로 띄움(독립 프로세스) → 그 프로세스 종료를 감시(watchdog)하다가 닫히면 7862 서버를 `taskkill /T /F` 로 종료.
- **확인:** 실측 — 앱 창 닫자 약 2초 내 서버 종료(포트 닫힘, python 0개).

## 6단계 — "브금만 만들기"에 보컬이 섞이던 문제
- **무엇을:** 브금만 모드인데 결과에 보컬이 들어가는지 확인 + 수정.
- **발견:** 합성 코드가 항상 `악기+보컬` 로 합침(최근 수정에서 빠진 회귀). 게다가 보컬 트랙을 똑같이 다 생성/후처리.
- **수정(`gradio_server.py`):** ① 브금만이면 stage2·보코더에서 **보컬 트랙(vtrack) 제외**, ② 16k/44k 합성 모두 **악기만** 출력, ③ stage2 라벨을 파일명 기준으로 정정.
- **확인(실측):** 브금만 생성 결과가 `_itrack.mp3`(악기), `vocoder/stems/vtrack.mp3` 갱신 안 됨(=보컬 미처리). (백업: `gradio_server.py.bak7_brgm_*`)

## 7단계 — 속도(GPU) 최적화 시도 → 되돌림
- **무엇을:** "너무 느림" 해결 시도. profile 3 가 모델을 RAM에 올려 GPU를 거의 안 쓰는 것 확인.
- **시도:** profile 3 에 VRAM 예산(budget) 부여 → 양자화 모델 VRAM 상주(생성 중 GPU 95% 사용).
- **결과:** **속도는 거의 그대로**(16GB라 int8 양자화 강제 → int8 연산 자체가 ~8 token/s 병목). 긴 곡에선 메모리 위험.
- **결정:** **원래 설정(profile 3 기본)으로 복원.** 속도 한계는 하드웨어/모델 문제로 결론.
- **핵심 사실:** YuE는 stage1에서 보컬·악기를 함께 생성하므로 "브금만"이어도 가장 큰 stage1 시간은 동일 → 전체가 크게 안 빨라짐.
